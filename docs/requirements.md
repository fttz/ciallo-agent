# 智能聊天助手需求文档

## 1. 文档目标

本文档用于定义一款面向普通个人用户的智能聊天助手 Web 应用需求。产品形态参考豆包、ChatGPT、通义千问、元宝等通用 AI 助手应用，但不走 Coze 一类“面向搭建者”的复杂平台路线。

本产品目标是提供一个开箱即用、界面简洁、能力完整的智能助手，让用户在尽量少配置、少学习成本的前提下，完成日常问答、内容创作、图片理解、文件解析、多模型切换等任务。

## 2. 产品定位

### 2.1 目标用户

- 普通个人用户
- 有日常 AI 对话、写作、总结、翻译、学习、办公辅助需求的用户
- 不希望面对过多参数、工作流、节点编排、工具细节的用户
- 有上传图片、文档、表格、演示文件并让 AI 分析内容需求的用户

### 2.2 产品价值

- 统一入口：在一个应用中接入多个模型提供商和多种模型能力
- 简洁易用：将复杂模型配置和能力选择收敛为清晰的用户体验
- 多模态支持：支持文本、图片、文件输入与解析
- 连续对话：围绕会话进行上下文管理和多轮交流
- 可扩展：后端可持续接入新的模型、文件解析器和业务能力

## 3. 产品目标

### 3.1 核心目标

- 提供一个稳定、流畅的 Web 聊天界面
- 支持接入多个大模型与多模态模型
- 支持用户上传图片与常见文件，并对内容进行解析
- 支持模型地址、API Key、模型名称等配置管理
- 支持会话管理、历史记录和消息流式输出
- 通过较少但有效的设置满足大多数普通用户需求

### 3.2 非目标

- 不做复杂智能体编排平台
- 不面向开发者暴露过多系统 Prompt、工具链编排、节点编排能力
- 不在第一阶段构建企业级组织权限、审批流、多租户运营后台
- 不在第一阶段做插件市场或开放生态平台

## 4. 典型使用场景

- 日常问答：用户像使用 ChatGPT 一样直接发起对话
- 写作创作：生成文章、润色文案、总结长文本
- 学习辅助：解释概念、整理知识点、生成提纲
- 办公辅助：解析 PDF、Word、Excel、PPT 并提炼重点
- 图片理解：上传图片，让模型描述、识别、问答
- 多模型切换：不同问题选择不同模型，如快速模型、深度推理模型、多模态模型
- 文件问答：基于上传文件内容继续追问与多轮对话

## 5. 功能需求

## 5.1 用户端功能

### 5.1.1 聊天界面

- 提供单页式或应用式聊天界面
- 支持消息列表展示用户消息和助手消息
- 支持 Markdown 渲染
- 支持代码块渲染、复制
- 支持表格、列表、引用等富文本内容显示
- 支持模型响应流式输出
- 支持中断生成
- 支持重新生成上一条回答
- 支持针对单条回答继续追问

### 5.1.2 会话管理

- 支持新建会话
- 支持查看历史会话列表
- 支持会话重命名
- 支持删除会话
- 支持按时间展示会话
- 支持在会话内保存完整消息上下文

### 5.1.3 模型选择

- 支持在聊天时选择可用模型
- 支持区分文本模型和多模态模型
- 支持展示模型的能力标签，如文本、视觉、快速、深度思考
- 支持配置默认模型
- 当某条消息附带图片时，前端只展示可用的视觉模型或由后端自动校验

### 5.1.4 文件与图片上传

- 支持上传图片文件
- 支持上传文档文件
- 支持拖拽上传
- 支持多文件上传
- 支持展示上传进度
- 支持展示上传成功、失败状态
- 支持删除已上传附件

### 5.1.5 文件解析与问答

- 系统能够解析用户上传的常见文件内容
- 解析后可将内容作为上下文提供给模型
- 用户可直接对文件内容提问
- 支持在同一会话中对文件进行连续追问
- 对超长文件应采用分段提取、摘要、检索或截断策略

### 5.1.6 应用设置

- 提供简洁设置页面
- 支持配置模型服务地址
- 支持配置 API Key
- 支持配置模型列表与模型别名
- 支持设置默认模型
- 支持设置是否保留聊天记录
- 支持设置界面语言与基础偏好项

## 5.2 管理与系统配置功能

### 5.2.1 模型配置管理

- 支持配置多个模型提供商
- 每个提供商至少包含名称、协议类型、Base URL、API Key、模型列表、是否启用
- 支持 OpenAI 兼容协议作为基础接入方式
- 支持后续扩展到 Anthropic、Gemini 或厂商专有协议
- 支持配置文本模型、视觉模型、音频模型等类别

### 5.2.2 文件解析能力管理

- 支持针对不同文件类型选择不同解析器
- 记录解析状态、解析耗时、失败原因
- 支持限制单文件大小、单次上传文件数量、支持格式白名单

### 5.2.3 运行监控

- 记录请求日志
- 记录模型调用耗时、成功率、错误率
- 记录上传与解析任务状态
- 支持基础告警能力

## 6. 支持的输入类型

### 6.1 文本输入

- 普通文本
- 多轮上下文消息
- 长文本粘贴

### 6.2 图片输入

- jpg
- jpeg
- png
- webp
- gif（首阶段可按静态图处理）

### 6.3 文件输入

- pdf
- docx
- txt
- md
- pptx
- xlsx
- csv
- json

### 6.4 后续可扩展输入

- 音频
- 视频截图或视频文件摘要
- 网页链接抓取

## 7. 文件解析需求

### 7.1 基础解析能力

- PDF：提取文本、页码结构、基础图片信息
- DOCX：提取标题、段落、表格
- PPTX：提取页面文本、备注、结构
- XLSX/CSV：提取表格数据、工作表结构，并生成适合问答的文本表示
- TXT/MD：保留原始文本结构
- JSON：格式化展示并转为可摘要内容

### 7.2 图片理解能力

- 支持上传图片后直接向视觉模型提问
- 支持图片描述
- 支持图片内容问答
- 后续可扩展 OCR 与图表理解

### 7.3 解析策略

- 小文件：可直接全文注入上下文
- 中等文件：优先生成结构化摘要并保留原文切片
- 大文件：采用切片、摘要、检索增强问答策略

## 8. 用户体验需求

- 界面应接近成熟 AI 助手产品，干净、稳定、易理解
- 不向普通用户暴露过多参数，如 temperature、top_p、上下文窗口等复杂概念
- 模型切换逻辑尽量自然
- 上传文件和图片的入口明显
- 首屏提供示例问题与能力提示
- 移动端和桌面端均可使用
- 聊天响应需要有明确的加载、流式输出和失败重试反馈

## 9. 非功能需求

### 9.1 性能

- 首屏加载速度可接受
- 消息发送后应尽快开始流式返回
- 文件上传与解析应异步化，避免阻塞主聊天流程

### 9.2 稳定性

- 模型调用失败时，前端有明确错误提示
- 文件解析失败时，可提示重新上传或更换格式
- 长时间任务应有状态反馈

### 9.3 安全性

- API Key 必须加密存储或至少安全保存，不在前端明文暴露
- 限制文件类型、文件大小、防止恶意上传
- 做好基础鉴权、限流、防滥用
- 敏感日志脱敏

### 9.4 可扩展性

- 新增模型提供商时尽量不影响现有接口
- 新增文件解析器时有统一抽象层
- 系统应支持后续接入知识库、搜索、语音等能力

## 10. 核心业务流程

### 10.1 纯文本聊天流程

1. 用户进入聊天页
2. 选择模型或使用默认模型
3. 输入问题并发送
4. 前端调用后端聊天接口
5. 后端根据模型配置路由请求
6. 模型返回流式结果
7. 前端实时展示回答并保存会话记录

### 10.2 图片问答流程

1. 用户上传图片
2. 前端校验格式和大小
3. 文件上传至对象存储或后端临时存储
4. 用户输入问题并发送
5. 后端确认当前模型支持视觉能力
6. 后端将图片和文本一并发送给多模态模型
7. 前端流式展示结果

### 10.3 文件解析问答流程

1. 用户上传文件
2. 系统异步解析文件
3. 解析结果保存为结构化文本和元数据
4. 用户围绕该文件提问
5. 后端按文件大小和上下文策略组织提示词
6. 必要时使用摘要或检索片段送给模型
7. 返回回答并保留引用上下文信息

## 11. 建议的信息架构

- 登录页（如首阶段需要用户体系）
- 聊天主页
- 历史会话侧边栏
- 模型选择器
- 文件上传区
- 设置页
- 个人中心页（可后置）

## 12. MVP 范围

### 12.1 第一阶段必须具备

- Web 聊天页面
- 多会话管理
- 至少支持 2 类模型接入方式
- 支持 OpenAI 兼容接口配置
- 支持文本聊天流式输出
- 支持图片上传和视觉问答
- 支持 PDF、DOCX、TXT、MD、XLSX、PPTX、CSV 基础解析
- 支持文件上传后继续提问
- 支持基础设置页
- 支持基础鉴权

### 12.2 第二阶段增强

- OCR
- RAG 检索增强
- 多文件联合问答
- 语音输入输出
- 移动端体验优化
- 分享会话
- 消息引用来源展示

### 12.3 第三阶段扩展

- Agent 能力编排
- 插件或工具调用
- 联网搜索
- 个人知识库
- 多设备同步与多端应用

## 13. 角色与权限建议

### 13.1 首阶段角色

- 普通用户
- 管理员

### 13.2 权限范围

- 普通用户：聊天、上传文件、查看历史、修改个人设置
- 管理员：管理模型配置、查看系统运行状态、查看错误日志、调整上传限制

## 14. 关键数据对象

- 用户
- 会话
- 消息
- 模型提供商配置
- 模型定义
- 上传文件
- 文件解析结果
- 模型调用日志

## 15. 风险与约束

- 不同模型供应商协议存在差异，统一抽象层必须提前设计
- 多模态能力在不同模型间差异较大，前端不能假设所有模型都支持图片
- 大文件直接喂给模型成本高且效果不稳定，需要摘要与检索策略
- 文件解析质量会显著影响问答体验
- 若 API Key 由用户自行配置，安全存储与权限隔离需要明确设计

## 16. 待确认问题

- 产品首阶段是否需要注册登录，还是仅做单用户部署版本
- 模型配置是仅管理员可配，还是允许用户在个人设置中配置自己的 API Key
- 是否需要计费、配额、调用次数限制
- 文件上传后保存时长与删除策略如何定义
- 是否需要公开分享链接功能
- 是否需要移动端优先适配或 PWA 支持

## 17. 验收标准

- 用户可以完成文本聊天、多轮对话和历史查看
- 用户可以切换模型并获得正常返回
- 用户可以上传图片并得到视觉模型回答
- 用户可以上传至少一种文档并围绕文档追问
- 模型地址、API Key、模型列表可以在设置中配置
- 系统在模型失败、网络失败、文件解析失败时有明确反馈
- 普通用户不需要理解底层模型参数即可完成核心任务
